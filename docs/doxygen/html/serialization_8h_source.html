00064     /// @page Serialization
00065     /// @section SerializationMain
00066     /// Serialization is the process of converting a class instance into a serial sequence of bits. DeSerialization is taking those
00067     /// bits and reconstructing th e original object without losing anything of value (in practice the only things not saved are cached
00068     /// values or values calulated as they are needed). These bits could be just about anything, because of its ubiquity we chose
00069     /// to serialize to xml (or something so similar as to be indistinguishable from standard xml). This allows a wide variety of tools
00070     /// to be used when working with and verifying these serialized classes. Additionally, transmitting and storing xml is easy to do,
00071     /// and can be done with a variety of other factors in mind. The xml text can be sent down any stream, put in any file, compressed,
00072     /// queried. You should see @ref XMLManual for information about the xml system itself.
00073     /// \n \n
00074     /// Topics:
00075     ///     - @ref serializationxml
00076     ///     - @ref serializationintegration
00077     ///         - @ref serializationmaking
00078     ///         - @ref serializationserializers
00079     ///         - @ref serializationlegacy
00080     ///     - @ref serializationoperators
00081     ///     - @ref serializationmisc
00082     /// @subsection serializationxml Serialization and XML
00083     /// The process of serializing doesn't just convert from class instance to text. Since our end goal is to convert live objects to
00084     /// XML it only make sense to closely integrate with the Mezzanine::xml portion of the engine. If you plan on writing serialization
00085     /// and deserialization code you should read the following parts of the @ref XMLManual at a minimum:
00086     ///     - @ref XMLDOM
00087     ///     - @ref XMLAccessingBasics
00088     ///     - @ref XMLModifyingNodeData
00089     ///     - @ref XMLModifyingAttributeData
00090     ///
00091     /// The central object that will carry information during this process is the Mezzanine::xml::Node. The Mezzanine::xml::Node is an excellent
00092     /// tool for converting and storing data in a single unified heirarchy.
00093     /// \n \n
00094     /// C++ and most object oriented languages heavily imply that class inheritance should be structured as hierarchies. Additionally
00095     /// Hierarchies are implied when complex class have other complex class or datatypes as members. Both of these structures map
00096     /// cleanly onto the kind of hierarchies that a well formed xml documents provide.
00097     /// \n \n
00098     /// There are some relationships in video game objects that cross normal hierarchical boundaries. For example, A constraint
00099     /// references two actors, and defines a relationship between them. When serialized the constraint simply stores the name of
00100     /// the actor and looks it up in the actor manager during deserialization. This implies that the actors exist already, or that
00101     /// there is some mechanism to create the constraint and fill in the actor later. Several mechanisms were discussed to
00102     /// accomplish this, some include: two passes of processing were constraintw would be done in the second pass, a work queue that
00103     /// would store objects that couldn't be deserialized yet, a prefetcher that would dig through the xml to find the required
00104     /// object.
00105     /// \n \n
00106     /// Those methods all likely could have been made to work. However, they are not optimal for a variety of reasons. All of them
00107     /// have a set of preconditions and require more computing resources and could potentially delay loading or transmission times.
00108     /// Some of them heavily imply that all of the items to deserialize must be stored in the same xml source. Some demand access to
00109     /// xml that may not have been transmitted yet.
00110     /// \n \n
00111     /// The simplest, most performant way to work around the issues that cross-hierarchical relationships presented was to ignore
00112     /// them. More specifically, throw an exception if an object reference during deserialization is not present. Then we ask that
00113     /// programmers who write could that must store, transmit and reconstruct class instances be aware of the following preconditions
00114     /// So can produce their own solutions:
00115     ///     - CollisionShapes must come before Actors and AreaEffects
00116     ///     - WorldNodes must come before Actors, Light and ParticleEffects. (this is still work in progress).
00117     ///         - Currently WorldNodes try to find the objects that are attached to them, and the attached tries to find the world node. (if one does not exist, this silently fails)
00118     ///     - Actors must come before constraints.
00119     ///     - Actors may have a WorldNode inside them, if this is the case, then the actor must come before Lights and ParticleEffects Attached to it.
00120     ///     - Actors must be done before SoundSets
00121     ///     - Sounds must be done before SoundSet (Still in progress)
00122     ///
00123     /// The easyiest way to meet these conditions and not consume an inordinate amount of computing resources, is to pay attention
00124     /// to the order that items are serialized in. If a program serializes the worldnodes, then the actors, then everything  else
00125     /// it will have relatively little trouble making it work.
00126     /// @subsection serializationintegration Integrate Serialization into Your Code
00127     /// There several ways to interact with the current serialization system. One might have to create a class that can be
00128     /// serialized or deserialized. There may be situations where another system is emitting xml and it must be intergrated into
00129     /// an existing game. It may be desired to create a 'factory' that produces objects from and xml source or create a sink to put
00130     /// objects into so they can be serialized. Here we will discuss some of the ways that the serialization system can be extended
00131     /// and what kind of assumptions it makes, so that anyone can write software that interacts with it cleanly.
00132     /// @subsubsection serializationmaking Make a Serializable or a DeSerializable
00133     /// Creating a class that be serialized is easy. There is just one function that it must implement. If a class implements this,
00134     /// it is said to be Serializable:
00135     /// @code
00136     /// void SerializableClass::ProtoSerialize(xml::Node&) const;
00137     /// @endcode
00138     /// The member ProtoSerialize(xml::Node&) is expected to accept a Mezzanine::xml::Node and attach exactly one Node to it. This new Serialized
00139     /// node should contain all the data stored in the current state of the object being serialized. Storing data outside of this
00140     /// one node could cause undefined behavior.
00141     /// \n \n
00142     /// The exact layout of the data in the Serialized Node is not pre-determined. The creator of that function need only
00143     /// take into account any difficulties DeSerializing when creating this. Because of this concern it is advisable name the Serialized
00144     /// node something unique and appropriate and to include a 'Version' attribute on it. If the class
00145     /// changes, the DeSerialization function will only need to check the 'Version' attribute to know if and how it can handle it.
00146     /// \n \n
00147     /// Integrating with the DeSerialization code is pretty easy too. There are two functions you are expected to implement to
00148     /// create a DeSerializable:
00149     /// @code
00150     /// void DeSerializableClass::ProtoDeSerialize(const xml::Node&);
00151     /// static String DeSerializableClass::SerializableName();
00152     /// @endcode
00153     /// The SerializableName() is expected to simply return the name of the xml elements this class will DeSerialize. For example
00154     /// A Mezzanine::Vector3 returns "Vector3", and a Mezzanine::ActorRigid return "ActorRigid". If a class is both DeSerializable and
00155     /// serializable it makes sense to call this function when assigning the name to the Serialized Node it creates.
00156     /// \n \n
00157     /// ProtoDeSerialize(const xml::Node&), accepts a Mezzanine::xml::Node. The Node passed to it would correspond to the Serialized
00158     /// Node created by the ProtoSerialize(xml::Node&) function listed above. If xml is created by something then this is calling
00159     /// code is expecting this function to be the correct deserialization function. It is advisable but not required to verify the
00160     /// name of the xml node matches what is expected and that
00161     /// the 'Version' is something this code can handle. It is also advisable that every piece of data pulled out is verified the
00162     /// best it can be. If exceptions are thrown for every discrepency, then programmers using this will create xml and code that
00163     /// produce no discrepencies.
00164     /// \n \n
00165     /// The following template make use of only the 3 functions described above to Serialize or DeSerialize class instances:
00166     /// @code
00167     /// template <class T> std::ostream& Serialize(std::ostream& Stream, const T& Converted, const String& Indent = String("") );
00168     /// template <class T> std::istream& DeSerialize(std::istream& Stream, T& Converted);
00169     /// @endcode
00170     /// The functions make calls on the Mezzanine::xml system and expect a fairly basic set of conditions to be met before they are used.
00171     /// Serialize accepts an output stream and the class instance to be Serialized. It will create an xml::Document and populate it
00172     /// data from the class provided and then emit that into the stream. DeSerialize accepts an inputstream and the object to be
00173     /// populated. It expects the next xml element in the stream to be a serialized version of the passed object and will then
00174     /// overwrite as many of the values of the passed object as possible with the serialized values. For small items DeSerialize
00175     /// is fine, where possible it is better to have the xml::Document open the file or stream itself as to prevent the second
00176     /// pass through to find exactly one xml element.
00177     /// \n \n
00178     /// @subsubsection serializationserializers Working with Serializers and Deserializers
00179     /// In some cases, there are some pieces of information that cannot be supplied or
00180     /// entered by the class itself. This data must be provided by another class or upon creation of the class. This other class
00181     /// can implement the Serializer, DeSerializer, or both interfaces to make working with large amounts of serialization easier.
00182     /// \n \n
00183     /// For example actors can only accept a mesh upon construction. So overwriting an existing actor is impossible to do completely.
00184     /// It expected to be partially implemented, to the extent possible, in the class members. But if you have the need to create
00185     /// Actors on the fly from data stored in files it makes sense to have a dedicated class or interface than can create these.
00186     /// Here is what goes into a Serializer:
00187     /// @code
00188     /// template <class Serializable> class Serializer
00189     /// {
00190     ///     virtual void Serializer::ProtoSerializeAll(xml::Node& CurrentRoot) const = 0;
00191     ///     virtual std::ostream& Serializer::SerializeAll(std::ostream& Stream) const;
00192     ///     virtual void Serializer::ProtoSerialize(const Serializable& Target, xml::Node& CurrentRoot) = 0;
00193     ///     virtual std::ostream& Serializer::Serialize(std::ostream& Stream, const Serializable& Target)
00194     /// };
00195     /// @endcode
00196     /// Serializer::ProtoSerialize() when implement should take the required steps to attach a Serialized Node to the
00197     /// Passed xml::Node that represent the Serialization target. It is expected to get the extra information that the target
00198     /// cannot provide from somewhere else. Ideally the the Serializer can be, or be associated with, a manager or container
00199     /// of some kind. There is not default implementation of this.
00200     /// \n \n
00201     /// Serializer::Serialize() Goes one step further than Serializer::ProtoSerialize() and also sends it down a stream. The
00202     /// default implements use Serializer::ProtoSerialize().
00203     /// \n \n
00204     /// Serializer::ProtoSerializeAll() performs a similar role to Serializer::ProtoSerialize(), but again, it goes one step
00205     /// further. Rather than accept a single Target to serialize it is expected that the Serializer go to the source of the
00206     /// Targets and serialize all of them that are available. All of the target should be contained in one Node attached to
00207     /// the Node the function accepts. This is not implemented by default, the logic is too specific to the items to be
00208     /// serialized.
00209     /// \n \n
00210     /// Serializer::SerializeAll() uses Serializer::ProtoSerializeAll() to send all of the available Targets in Serialized
00211     /// down a stream.
00212     /// \n \n
00213     /// The logic behind a DeSerializer is similar to a Serializer. The same types of methods, even similar implementations
00214     /// if the function is implemented. Like the ProtoDeSerialize() individual DeSerializables implement, the functions on
00215     /// a DeSerializer will be passed the nodes that would correspond to the those created by their counterparts on the
00216     /// Serializer. Here is the contents of a Deserializer:
00217     /// @code
00218     /// template <class DeSerializable> class DeSerializer
00219     /// {
00220     ///     virtual void DeSerializer::ProtoDeSerializeAll(const xml::Node& OneNode) = 0;
00221     ///     virtual std::istream& DeSerializer::DeSerializeAll(std::istream& Stream)
00222     ///     virtual DeSerializable* DeSerializer::ProtoDeSerialize(const xml::Node& OneNode) = 0;
00223     ///     virtual std::istream& DeSerializer::DeSerialize(std::istream& Stream)
00224     ///     virtual String ContainerName() const = 0;
00225     /// };
00226     /// @endcode
00227     /// The function ContainerName() should be used when creating and verifying the xml element that is parent to the items
00228     /// DeSerialized by ProtoDeSerializeAll(). The Default implmentation of DeSerializeAll() will use ContainerName to
00229     /// verify it has extracted the correct Node.
00230     /// \n \n
00231     /// There is no technical reason why a class cannot be both a serializer and a deserializer, or even multiple kinds of
00232     /// Serializers or DeSerializers. To keep things simple the Managers provided by the Mezzanine engine will store a pointer
00233     /// to the appropriate Serializer when one is required.
00234     /// @subsubsection serializationlegacy Integrating with External XML Providers
00235     /// Sometimes yu will be forced to work with a system that produces xml that is not structured in a similar way to this
00236     /// system. Sometimes it may be too costly or not possible to modify the code to integrate it. For these the following
00237     /// function exists:
00238     /// @code
00239     /// template <class T> void SloppyProtoSerialize(const T& Converted, xml::Node& CurrentRoot)
00240     /// @endcode
00241     /// This function will make a call on the the stream insertion operator of the class passed in. If one doesn't exist
00242     /// it is easy to add one in your code without chaning the original source. If one does exist than you should probably
00243     /// copy/paste the whole function and re-implement it calling the functions that emit the XML string or stream. If you
00244     /// want to implement a stream insertion operator, the function prototype should be similar to the stream insertion
00245     /// operator in the @ref serializationoperators section.
00246     /// @subsection serializationoperators Serialization Operators
00247     /// The stream insertion (<<) and stream extraction (>>) operators can be used for serializing and deserializing most items
00248     /// in the Mezzanine engine.
00249     ///
00250     /// Unfortunately due to conflict with the stream insertion operators provided with the iostreams library these couldn't be
00251     /// made into a template. That doesn't mean that they are difficult to implemt. Here is a typical implemenation of stream
00252     /// insertion operators for XML serialization:
00253     /// @code
00254     /// #ifdef MEZZXML
00255     ///
00256     /// std::ostream& operator << (std::ostream& stream, const Mezzanine::ActorRigid& ActorToSerialize)
00257     /// {
00258     ///     Serialize(stream, ActorToSerialize);
00259     ///     return stream;
00260     /// }
00261     ///
00262     /// std::istream& operator >> (std::istream& stream, Mezzanine::ActorRigid& x)
00263     ///     { return DeSerialize(stream, x); }
00264     ///
00265     /// void operator >> (const Mezzanine::xml::Node& OneNode, Mezzanine::ActorRigid& x)
00266     ///     { x.ProtoDeSerialize(OneNode); }
00267     ///
00268     /// #endif  // \Mezzxml
00269     /// @endcode
00270     /// You will want to implement these functions with the appropriate type. The type Mezzanine::ActorRigid is used purely as example
00271     /// Though this is actual working code and was in the engine at one point.
00272     /// The "#ifdef MEZZXML" Tells the compiler to skip this chunk of code if MEZZXML is not enable. Since this is a configuration
00273     /// option of the engine you may wish for you code to be aware of it. More sophisticate handling of this is possible, for example
00274     /// The Mezzanine::Vector3 emits a string in the form of [X,Y,Z], if Mezzxml is not enabled.
00275     /// \n \n
00276     /// The function operator<< simply calls Serialize and returns the stream, so it has all the pre and cost conditions of the Serialize
00277     /// function listed in the @ref serializationmaking section.
00278     /// \n \n
00279     /// The stream extraction operators are a little bit more interesting. The operator>>(istream,YourType), by virtue of calling Deserialize
00280     /// will wind up taking two passes over the XML. One looking for the ending tag that matches the first (it gets all the children of that tag too)
00281     /// and one performing the actual parsing. The operator>>(istream,YourType) will work only with completely parsed objects in memory. With the
00282     /// combination of these two all the heavy lifting of parsing is done up front, and the rest of the deserialization is just a bunch of pointer
00283     /// and string manipulation. Another possibility with your stream extraction operator, if you new that it had exactly one parent xml node ,you
00284     /// create without that first pass for improved performance.
00285     /// @subsection serializationmisc Other little Things
00286     /// To simplify and standardize errors thrown, the following functions exist:
00287     /// @code
00288     /// void SerializeError(const String& FailedTo, const String& ClassName, bool SOrD = true);
00289     /// void DeSerializeError(const String& FailedTo, const String& ClassName, bool SOrD = false);
00290     /// @endcode
00291     /// Both of these functions throw a Mezzanine::Exception with the descriptive text of "Could not {FailedTo} during {ClassName} [De]Serialization."
00292     /// If SOrD (Serialize Or Deserialize) is true the "De" is not printed.

00358     /// @brief A tool for deserializing classes with specific issues deserializing them
00359     /// @details Some classes Must have certain values available at the time of construction. This make deserializing them by overwriting an existing
00360     /// class instance impractical. \n \n
00361     /// This is expected to work with classes that have implemented the required DeSerializable functions. Specifically This makes use of
00362     /// "static String SerializableName()", and it is expected that functions that must be implemented would call on "void ProtoDeSerialize(const xml::Node&)".
00363     /// The type of this template is expected to match what this is deserializing.
00364     /// \n \n

00366     /// no reason why something could not inherit from this and Mezzanine::Serializer.

00368     template <class DeSerializable>
00369     class DeSerializer
00370     {
00371         #ifdef MEZZXML
00372         public:
00373         /// @brief Convert An XML Node into a complete series of live class instances
00374         /// @param OneNode A reference to the XML node to reconstitute into multiple Live classes.
00375         /// @details This is expected to put the deserialized items somewhere they can be accessed by the calling,
00376         /// but provides no facility for working them itself. \n \n
00377         /// Not implemented in default DeSerializer.
00378         virtual void ProtoDeSerializeAll(const xml::Node& OneNode)
00379         {
00380             // no checking occurs here, because this should be DeSerializeAll(istream&)
00381             xml::Node SingleItemNode = OneNode.GetFirstChild();
00382             while(SingleItemNode)
00383             {
00384                 ProtoDeSerialize(SingleItemNode);
00385                 SingleItemNode = SingleItemNode.GetNextSibling();
00386             }
00387         }
00388         /// @brief Get One node that has several of the appropriate kinds of nodes as children and deserialize all of them
00389         /// @param Stream The std::istream to get the data from.
00390         /// @details The default implementation of this uses ProtoDeSerializeAll(xml::Node&) to accept11
00391         /// The complete XML to serialise and assemble it in memory.
00392         /// @return This returns the input stream after the xml document has been extracted from it.
00393         virtual std::istream& DeSerializeAll(std::istream& Stream)
00394         {
00395             Mezzanine::String OneTag( Mezzanine::xml::GetOneTag(Stream) );
00396             std::auto_ptr<Mezzanine::xml::Document> Doc(Mezzanine::xml::PreParseClassFromSingleTag(this->ContainerName(), OneTag) );
00397             ProtoDeSerializeAll(Doc->GetFirstChild());
00398             return Stream;
00399         }
00400 
00401         /// @brief Convert An XML Node into a complete live data structure
00402         /// @param OneNode A reference to the XML node to reconstitute into a live class instance.
00403         /// @details Not implemented in default serializer.
00404         /// @return A pointer to the freshly deserialized class instance.
00405         virtual DeSerializable* ProtoDeSerialize(const xml::Node& OneNode) = 0;
00406         /// @brief Get the serialized version of all the live data from the stream.
00407         /// @param Stream The std::istream to get the data from.
00408         /// @details The default implementation of this uses ProtoDeSerializeAll(xml::Node*) to accept
00409         /// The complete XML to serialise and assemble it in memory.
00410         /// @return This returns the input stream after the xml document has been extracted from it.
00411         virtual std::istream& DeSerialize(std::istream& Stream)
00412         {
00413             Mezzanine::String OneTag( Mezzanine::xml::GetOneTag(Stream) );
00414             std::auto_ptr<Mezzanine::xml::Document> Doc(Mezzanine::xml::PreParseClassFromSingleTag(DeSerializable::SerializableName(), OneTag) );
00415             ProtoDeSerialize(Doc->GetFirstChild());
00416             return Stream;
00417         }
00418 
00419         /// @brief This will return the Name of the element that Contains multiple of the items to be DeSerialized
00420         /// @return A String that correctly indicates the name of an xml tag.
00421         virtual String ContainerName() const = 0;
00422         #endif
00423     };
