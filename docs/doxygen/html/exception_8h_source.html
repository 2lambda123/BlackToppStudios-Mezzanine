2887 /// @brief An easy way to throw exception with rich information.
2888 /// @details An important part of troubleshooting errors from the users perspective is being able to
2889 /// tie a specific 'fix' to a specific error message. An important part of that is catching the
2890 /// right exceptions at the right time. It is also important to not allocate more memory or other
2891 /// resources while creating an exception.
2892 /// @n @n
2893 /// This macro makes doing all of these easy. Every exception thrown by this macro with provide the
2894 /// function name, the file name and the line in the file from which it was thrown. That provides
2895 /// all the information the developer needs to identify the issue. This uses some specific template
2896 /// machinery to generate specifically typed exceptions static instances at compile to insure the
2897 /// behaviors a programmer needs. Since these are allocated (optimized out really) when the program
2898 /// is first loaded so there will be no allocations when this is called, and the type is controlled
2899 /// by the error number parameter.
2900 /// @n @n
2901 /// As long as the automation developer provides a unique string for each failure, then any messages
2902 /// logged or presented to the user or log will uniquely identify that specific problem. This
2903 /// allows the user to perform very specific web searches and potentially allows
2904 /// troubleshooters/technicians to skip lengthy diagnostics steps.
2905 /// @param num A specific code from the @ref ExceptionBase::ExceptionCodes enum will control the
2906 /// type of exception produced.
2907 /// @param desc A message/description to be passed through to the exceptions constructor.
2908 #define MEZZ_EXCEPTION(num, desc) throw ExceptionFactory<num>::Type(desc, __func__, __FILE__, __LINE__ );
2909 #endif
