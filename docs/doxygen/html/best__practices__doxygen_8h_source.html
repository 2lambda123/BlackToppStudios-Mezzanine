44  @page best_practices_doxygen Best Practices â Doxygen
45  Doxygen is a tool for extracting API documentation from source code. Doxygen reads source code
46  the same way a C, C++ or Java compiler would, but can also read special comments to include
47  extra information. This page is about making the most of doxygen by entering high quality
48  information. If you want to learn how to run doxygen see the page on @ref running_doxygen or if
49  you want to know how to install it see the appropriate page about our @ref dev_tools_menu.

51  @section best_practices_doxygen_visibility D1 Improve Errors visibilty during builds
52  Errors can't be fixed if you cannot see them. Try to use tools that will show you the errors
53  that running the build process creates. Using.... describe more about IDEs here.

55  @section best_practices_doxygen_minimum D2 Minimum Amount to Document
56  It is easy to document too much or too little. Too much and it will easily fall out of date when
57  the code changes and developers ignore the wall of text. Too little and people using the code
58  and documentation go WTF?! and use some other code or write from scratch.
59 
60  @n
61  Here is what we do. We probably tend to go a little too heavily, but we are writing an API and
62  we feel clarity is worth some maintanence cost.
63 
64  @subsection best_practices_doxygen_minimum_function Function/Methods Minimum Amount to Document
65  Each function/method should get:
66 
67  - \@brief Every function and method should get one of these unless the \@return or \@param
68  describe it so completely adding this would hurt. (leave \@brief off the int GetX() method on
69  2D point class for example).
70  - One \@param for each parameter describing its type, accepted values and any default values.
71  - \@return Unless it returns void then the return tty and value shoud be documented.
72  - \@throw If an exception is thrown, its type and conditions for causing the throw should be
73  noted.
74  - \@details Should be added anytime the function has special preconditions. changes state on
75  a class that is not obvious or is otherwise complex.
76 
77  Here are some examples of Bad and Good doxygen comments on bad code. This code is terrible to
78  highlight how much of a difference good and bad documentation can make. In a real situation it
79  it likely that even good code with good variable names might still be complex enough to allow
80  misconceptions in.
81 
82  @n
83  Bad Dox:
84  @code
85  int sqrt(int x);
86 
87  /// \@brief calculates absolute value
88  int abs(int x);
89 
90  /// \@param Accepts a float.
91  /// \@return Returns a float.
92  float log(float x)
93  @endcode
94 
95  Good Dox:
96  @code
97  /// \@param x An Integer to take the square root of
98  /// \@return An int containing the square root of x. Any floating part is truncated
99  int sqrt(int x);
100 
101  /// \@brief Calculates absolute value, the distance from zero
102  /// \@param x An int that might be postive or negative.
103  /// \@return A positive int (or 0) that is the same as x but with an minus sign removed.
104  int abs(int x);
105 
106  /// \@brief Calculate the natural logorithm of a value.
107  /// \@param x A float to calculate the log2 of.
108  /// \@return Returns a floatX.
109  float log(float x)
110  @endcode

161  @subsection best_practices_doxygen_minimum_class Class Minimum Amount to Document
162  Most classes should have an \@brief providing a simple overview, an \@details providing
163  additional rationle and perhaps a few examples in \@code/\@endcode or \@verbatim/\@endverbatim
164  sections if their use is complicated. Since every method should be well named and well
165  documented hopefully class documentation can be small.
166 
167  @n
168  Try to docuent classes in the header file just ahead of the class keyword and \@class will not
169  be needed.
170 
171  @n
172  Example:

177  /// @verbatim
178  /// Matrix UnitMatrix;

180  /// Matrix Product = UnitMatrix * OtherMatrix;
181  /// @endverbatim

187  @subsection best_practices_doxygen_minimum_file File Minimum Amount to Document
188  All Files get an \@file and an \@brief describing a little more than the name of the file could
189  desribe. This mostly provides a link in doxygen that might not otherwise have been generated.

200  @section best_practices_doxygen_copydoc D3 \@copydoc Should Follow inheritance
201  The \@copydoc directive should primarily be used by following up inheritance heiarchies. When
202  used to in other ways it can creates logical dependencines in the documentation that do not
203  match the code which increases future maintanence effort. Consider the follow inheritance tree
204  of classes:
205 
206  @dot Sample inheritance tree
207  digraph interop {
208  Animal [shape=box];
209  Mammal [shape=box];
210  Reptile [shape=box];
211  Cat [shape=box];
212  Dog [shape=box];
213  Lizard [shape=box];
214  Animal -> Mammal -> Cat;
215  Mammal -> Dog;
216  Animal -> Reptile -> Lizard;
217  }
218  @enddot
219 
220  @n
221  Because the Lizard and the Cat have little to do with each other in code they will change in
222  different in the future. Even if they each have an int LegCount() const member function that
223  return 4, they should not copy documentation from eachother. Someone updating the lizard to have
224  6 legs is unlikely to be concerned the Cat class.
225 
226  @n
227  A more reasonable solution is to have Lizard copy documentation from Reptile and Cat from
228  Mammal. This makes it easy for someone changing code and documentation to know what other
229  documentation should be updated. Following a rule like this also implies that someone changing
230  a class has some responsibility to verify the documentation in each derived class.
231 
232  @n
233  Bad:
234  @code
235  /// \@brief All lizards have 4 legs.
236  /// \@return An integer of value 4
237  int Lizard::LegCount() const;
238 
239  /// \@copydoc Lizard::LegCount() const
240  int Cat::LegCount() const;
241  @endcode
242 
243  @n
244  Good:
245  @code
246  /// \@return All Mammals have 4 limbs so this returns an integer of value 4.
247  int Mammal::LegCount() const;
248 
249  /// \@copydoc Mammal::LegCount() const
250  int Cat::LegCount() const;
251  @endcode

254  @section best_practices_doxygen_identifier D4 Identifier Commands
255  \@enum, \@typedef, \@class, and other specifiers of things to be documented should be used
256  when documenting things out of line. If the documentation comments precede what you are
257  documenting then these can be omitted. This makes documentation shorter and easier to maintain.
258 
259  @n
260  Some of our older and more stale dox have some of these dangling from when we did not know
261  better.

266  class, function, section or page name you will get a link with the text of the name text from
267  the refferred item. If you want different text include link text in quotes. See some examples:
